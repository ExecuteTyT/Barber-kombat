# Модуль: Админы — звонки, гостевой опрос (Яндекс Формы), KPI/геймификация

Статус: **дизайн согласован** (3 решения ниже), часть по опросу ждёт структуру формы.
Дата: 2026-06-03.

## Context
Заказчик хочет завести администраторов в MAKON с учётом их работы:
1. **Ежедневные звонки** — сколько выполнено (как в DataHeroes, но у того нет API → ведём у себя).
2. **Гостевой опрос** (Яндекс Форма `makon.men/guest-survey`) — содержит в т.ч. вопросы про
   администраторов; ответы нужно забирать и учитывать.
3. Итог — **KPI-контроль для владельца** и **геймификация/мотивация для админов** (как Kombat у барберов).

Согласованные решения:
- Звонки: **список «кому звонить» + отметка «позвонил»** → авто-% выполнения (данные наши, из YClients).
- Цель: **KPI-дашборд (владелец) + геймификация (админ)**.
- Опрос: **нативный вебхук Яндекс Формы → наш бэкенд** (подтверждено: Формы шлют POST на URL;
  нужен домен — есть `barber.braidgroup.ru`; ретраи 7×/30мин; таймаут 5с; ответ 200/201/202).

## Архитектура (3 под-модуля)

### A. Звонки администратора (независимо от опроса)
- Источник списка — YClients-данные, уже есть в `AdminService.get_tasks`
  ([admin.py](../../backend/app/services/admin.py)): неподтверждённые записи на завтра,
  «ушедшие»/незаполненные ДР. Расширяем: формируем **дневной call-list** по филиалу.
- Новая таблица `admin_call_logs` (или отметка на визите/клиенте): `admin_id`, `branch_id`,
  `client_id`/`yclients_record_id`, `date`, `called_at`, `result` (дозвонился/нет/перезвонить).
- Эндпоинт: `POST /admin/calls/{branch_id}/mark` — отметить звонок; `GET /admin/calls/{branch_id}` —
  список на сегодня + % выполнения. Branch-scope через `require_branch_access` (уже есть).
- Фронт: в админ-зоне экран/вкладка «Звонки» — список клиентов, кнопка «Позвонить» (`tel:` /
  `wa.me`, переиспользуем `lib/contact.ts`) + отметка «позвонил», прогресс дня.

### B. Гостевой опрос (Яндекс Формы → вебхук) — ГОТОВО, проверено

**Эндпоинт:** `POST /api/v1/webhooks/yandex-forms`. Авторизация — общий секрет
(`settings.yandex_forms_secret`) в заголовке `X-Survey-Secret` **или** поле `secret` в теле.
Всегда отвечает 200 (без retry-штормов). Хранит весь payload в `survey_responses.raw` + извлекает
и считает поля.

**Контракт (что Форма должна слать POST-ом, плоский JSON).** В Яндекс Форме:
Интеграции → API → «Запрос заданным методом» → POST на наш URL, тело с переменными формы:
```json
{
  "secret": "<общий секрет>",
  "phone": "{телефон}",
  "branch": "{филиал}",                 // текст варианта: «Менделеева 17Б…»/«Корабельная 53…»
  "stars": "{общие впечатления}",        // 1–5 / «★★★★★»
  "recommend": "{порекомендовали}",      // «Да…»/«Нет…»
  "comment": "{замечания}",
  "admin_communication": "{Q23 общение с админом}",   // Хорошо/Нормально/Плохо/Очень плохо
  "admin_greeting": "...", "admin_amenities": "...", "admin_drinks": "...",
  "admin_staff_greeting": "...", "admin_next_visit": "...", "admin_promo": "...",  // Да/Нет
  "master_quality": "{Q18 качество стрижки}",   // 6 уровней
  "master_return": "{Q19 вернётесь к мастеру}"  // Да/Не уверен/Нет
}
```
Маппинг (проверено live): `branch` Cyrillic ↔ Latin БД по номеру улицы (17/53); `phone` → клиент;
клиент+филиал → **барбер последнего визита**. Скоринг: `admin_score`/`master_score` 0–100,
`stars`, `recommend`, `is_negative` (звёзды≤2 / не рекомендует / «плохо» админ / «не понравилась».

### B (исходный план)
- Эндпоинт `POST /api/v1/webhooks/yandex-forms` (по образцу
  [webhooks.py](../../backend/app/api/webhooks.py) YClients): принимает JSON, проверяет секрет
  (заголовок/поле — настраивается в Форме), отвечает быстро 200.
- Контракт JSON **определяем мы**, заказчик маппит вопросы формы на наши поля в настройке
  интеграции. Минимум: `phone`, `branch` (Менделеева/Корабельная → branch_id), и ответы-оценки.
- Таблица `survey_responses`: `organization_id`, `branch_id`, `client_id` (по телефону), `phone`,
  `raw` (JSONB — весь ответ), извлечённые оценки (по мастеру/сервису/**админу**), `created_at`.
- Маппинг branch по тексту варианта; client по телефону (нормализация — `phoneDigits`/бэкенд).
- Негатив из опроса → можно вести как отзывы (лента/алярм) — переиспользуем инфраструктуру негатива.

### C. KPI + геймификация админов
- Агрегат по админу за период: **% звонков**, **оценка из опроса** (вопросы про админа),
  выполненные задачи (подтверждения и т.п.). Формула/веса — настраиваемые (по образцу
  `RatingConfig`/ПВР у барберов).
- **Владелец:** дашборд по админам (контроль). **Админ:** свой экран с баллами/местом (мотивация).
- Переиспользуем паттерны рейтинга барберов (`daily_ratings`/`RatingEngine`) как ориентир.

## Findings (проверено вживую на боевом YClients) — важно

При разработке вскрылись факты, меняющие подход к звонкам:
- **Синк не тянул будущие записи** (только сегодня/вчера) → звонить было не по чему, а
  «tomorrow»-метрики админа всегда 0. **Исправлено (Stage 0):** polling синкает `today .. +14 дней`.
- **У YClients-записи есть флаг `confirmed`** (подтверждение брони). **Захватываем его**
  (`Visit.confirmed`) — это объективный источник «звонки/подтверждения сделаны», без ручной
  отметки и без API DataHeroes.
- **Наблюдение:** 33 ближайшие записи Менделеева — **все confirmed (100%)**. DataHeroes/админы уже
  подтверждают всё. Значит список «неподтверждённых для обзвона» сейчас пуст, а ценность MAKON =
  **показывать % подтверждения** (KPI), а ручная отметка «позвонил» — подстраховка для редких
  неподтверждённых/реактивации.

Вывод по дизайну: KPI звонков = **% подтверждения ближайших записей** (из `Visit.confirmed`),
плюс опциональный ручной call-list для `confirmed=false`.

## Этапы реализации
0. **Sgtage 0 — данные (ГОТОВО, проверено):** синк будущих записей (`+14д`) + захват
   `Visit.confirmed` + миграция 0006 + тесты (507 passed). Боевая проверка: 33 upcoming, 100% confirmed.
1. **A. Звонки/подтверждения** — метрика % подтверждения в админ-метриках (чинит «tomorrow»-метрики)
   + `admin_call_logs` + ручная отметка для `confirmed=false` + админ-экран + тесты.
2. **B. Вебхук опроса** — endpoint + `survey_responses` (raw + phone/branch) + секрет + тесты;
   заказчик настраивает HTTP-интеграцию в Форме на наш URL.
3. **C. Извлечение оценок про админа + KPI + геймификация** — после получения структуры формы.
