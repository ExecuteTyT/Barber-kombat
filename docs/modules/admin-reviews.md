# Модуль: Админ-зона — отзывы и работа с клиентом

Статус: **спецификация утверждается** (дизайн согласован, до реализации).
Дата: 2026-05-31.

## Context / зачем
Отзывы клиентов салона лежат в **YClients** (подтверждено: `GET /api/v1/comments/{company_id}`
отдаёт rating, master_id, record_id, text, дату, телефон/имя клиента; данные свежие). Сейчас в
MAKON обработка отзывов живёт в остатках роли `chef` (которая убрана из API в `cc9b109`) и
фактически доступна только owner, а у роли `admin` своего экрана отзывов нет.

Заказчик хочет, чтобы **администраторы филиала** в отдельной зоне MAKON: видели отзывы (особенно
негатив), быстро их обрабатывали и **писали клиенту**. DataHeroes API не имеет, поэтому
интеграция — через общий YClients (чтение отзывов), а отправка сообщения — через композер +
`wa.me` (без своего канала, без конфликта с DataHeroes).

## Согласованные решения
- **Формат:** зона в текущем приложении (расширяем роль `admin`), не отдельный проект.
- **Охват:** админ видит **только свой филиал** (`user.branch_id`).
- **Отправка клиенту:** Вариант C — композер + 1 клик `wa.me` (+ кнопка звонка `tel:`). Своего
  канала рассылки MAKON не заводит.
- **Источник отзывов:** YClients (не DataHeroes, не свой публичный сбор как основной).

## Объём работ

### 1. Бэкенд — синк отзывов из YClients
- Добавить `get_comments(company_id)` + схему `YClientComment` в
  [yclients/client.py](../../backend/app/integrations/yclients/client.py) (по образцу `get_records`).
- Синк: маппинг `salon_id → branch` (`_resolve_branch`), `master_id → barber` (`_resolve_barber`),
  `record_id → visit` (по `yclients_record_id`), `rating`, `text → comment`, `date → created_at`.
- Идемпотентность: новое поле `reviews.yclients_comment_id` (миграция alembic) — не задваивать.
- `source = "yclients"`. Негатив (≤3) → `status=NEW` + `_publish_new_review`; позитив (4–5) →
  `PROCESSED` (как в текущем `create_review`).
- Отзывы без мэппинга на мастера — пропускаем с логом (модель требует `barber_id`); решение
  расширить схему — отдельно.
- Запуск: рядом с обычным sync-таском (по расписанию) для каждого активного филиала.

### 2. Бэкенд — доступ и уведомления
- **Фикс дыры доступа:** в `/admin/*` и review-эндпоинтах проверять
  `current_user.role == OWNER or current_user.branch_id == branch_id` — иначе 403. Сейчас админ
  может подставить чужой `branch_id`.
- **Подключить Telegram-алерт негатива:** задача `send_negative_review_alert` существует, но не
  вызывается — вызвать при создании негативного отзыва; адресат — конфиг уведомлений филиала.

### 3. Фронт — экран «Отзывы» в админ-зоне
- Новый таб в `screens/admin/` (рядом с Metrics/Tasks/History), `RoleGuard section="admin"`.
- Лента отзывов **своего филиала**: фильтры (все/негатив/необработанные), бейдж счётчика,
  реалтайм через WebSocket (`new_review`, фильтр по своему `branch_id`).
- Карточка отзыва → обработка: статус (in_progress/processed) + комментарий (переиспользовать
  логику `ReviewProcessModal`, перенеся её из `screens/chef/` в `screens/admin/`).
- **Композер ответа клиенту:** поле текста + шаблоны быстрых ответов → кнопка «Написать в
  WhatsApp» (`https://wa.me/<phone>?text=<encoded>`), кнопка «Позвонить» (`tel:<phone>`).
  Отправленный текст сохраняем в `processed_comment` (след решения).
- Контекст клиента: имя, телефон, история визитов, мастер.

### 4. Чистка
- Убрать мёртвые остатки роли `chef` из потока отзывов (`screens/chef/BranchScreen` review-часть,
  `ReviewProcessModal` переехал в admin). Enum `UserRole.CHEF/MANAGER` — пометить как deprecated
  (трогать осторожно, на них могут быть данные).
